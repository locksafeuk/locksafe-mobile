Bugs & Issues

CRITICAL BUG-01: OneSignal App ID Mismatch Between Source and Production AAB

Location: base/assets/app.config inside AAB vs .env, eas.json, app.config.js

Detail: The production AAB embeds OneSignal App ID os_v2_app_zum5e4ck... but every source file and build profile references cd19d270-4a74-4bdf-b534-3287cfb8b4e4. The runtime resolution chain in pushNotifications.ts reads Constants.expoConfig?.extra?.oneSignalAppId first, which resolves to the AAB-embedded value. This means the production build initialises OneSignal with a completely different App ID than what's configured in the repo and eas.json.

Impact: Push notifications may be sent to the wrong OneSignal project, or fail entirely. If the embedded ID is the correct one, then the repo config is stale and misleading. Either way, this discrepancy is a deployment blocker — it must be clarified which ID is canonical and all sources must be aligned.

CRITICAL BUG-02: No Authentication Guard on Locksmith Routes

Location: app/(locksmith)/_layout.tsx

Detail: The locksmith dashboard, jobs, earnings, and settings screens have zero authentication checks. Navigating directly to /(locksmith)/(tabs) renders the full dashboard without login. Verified on web — all four tabs (Dashboard, Jobs, Earnings, Settings) are fully accessible without any auth token.

Impact: On Android, deep links via locksafe:// scheme or push notification navigation could land unauthenticated users on protected screens. The dashboard renders with default/empty data ("Hello, Partner", 5.0 Rating) which could confuse users. API calls from these screens will fail with 401, but the UI is still exposed. On web, this is a direct security issue.

Fix: Add an auth guard in (locksmith)/_layout.tsx that checks useAuthStore and redirects to login if user is null.

HIGH BUG-11: Apple App Store Connect API Private Key Exposed on Disk

Location: /home/ubuntu/locksafe-mobile/api_key.json

Detail: Contains Apple App Store Connect API private key (Key ID: CN472U9F7K, Issuer: 7472da00). While not currently tracked in git, the file is not in .gitignore (only *.p8 and *.key are ignored). A single git add . would commit this key to the repository.

Impact: If committed, anyone with repo access could upload builds to the App Store, modify app metadata, or access App Store Connect API on behalf of the developer account.

Fix: Add api_key.json to .gitignore immediately. Consider rotating the key if the repo has been shared.

MEDIUM BUG-03: "Forgot Password?" Link is Non-Functional

Location: app/(auth)/locksmith-login.tsx

Detail: The "Forgot password?" element is a Pressable with no onPress handler. Tapping it does nothing on any platform.

Impact: Locksmiths who forget their password have no recovery path in the app. They would need to contact support or use the website.

MEDIUM BUG-04: Backend Missing Server-Side Validation for Registration

Location: POST /api/locksmiths/register

Detail: The backend does not validate password length or email format. A 3-character password ("abc") passes the password check — the only validation failure is "Base location is required". Similarly, "notanemail" is accepted as a valid email. Client-side validation catches these, but the API is unprotected against direct calls.

Impact: Weak passwords and invalid emails could be registered via direct API calls, bypassing the mobile app's client-side validation.

MEDIUM BUG-05: Registration Validation Uses Alert.alert() Only — No Inline Errors

Location: app/(auth)/locksmith-register.tsx

Detail: Registration validation errors (missing fields, password mismatch, weak password) use Alert.alert() exclusively. Unlike the login screen which has an inline error banner, registration has no visible error state in the UI. On web, these alerts may be blocked by popup blockers.

Impact: Inconsistent UX between login and registration. On web, users get no feedback at all when validation fails.

MEDIUM BUG-09: OneSignal Subscribe POST Endpoint Not Authenticated

Location: POST /api/onesignal/subscribe

Detail: While the security fixes correctly added auth to GET subscribe and POST send, the POST subscribe endpoint accepts unauthenticated requests. It returns 404 (user not found) for invalid users, but anyone who knows a valid locksmith userId could register a playerId for push notifications.

Impact: Potential for push notification hijacking if user IDs are enumerable or leaked.

MEDIUM BUG-10: Token Refresh Endpoint Does Not Exist

Location: POST /api/auth/refresh (backend) / services/api/client.ts (mobile)

Detail: The mobile API client implements a token refresh mechanism that calls /api/auth/refresh, but this endpoint returns the Next.js 404 page (HTML). The client handles this gracefully (silent failure), but token refresh is completely non-functional.

Impact: When auth tokens expire, users will be silently logged out instead of having their session refreshed. This could cause unexpected logouts during active use.

LOW BUG-06: iOS buildNumber Stale in AAB Embedded Config

Detail: The AAB's embedded app.config shows iOS buildNumber as "3", but the current app.config.js source has "4". This indicates the AAB was built before the iOS build number was bumped. No functional impact on Android, but suggests the AAB may not include the latest code changes.

LOW BUG-07: Duplicate Permissions in AAB Config

Detail: ACCESS_COARSE_LOCATION, ACCESS_FINE_LOCATION, and CAMERA are each declared twice in the AAB's embedded permission list. This is caused by plugin config merging and has no functional impact, but indicates sloppy configuration.

INFO BUG-08: Remember Me Toggle Not Visible on Web

Detail: The login screen's Remember Me section with Switch component doesn't render on the web version. This is likely a React Native Web rendering issue with the Switch component. On native Android, this would render correctly.

Positive Findings

✅ OneSignal Race Condition Fix Verified: The lazy initialization pattern in _layout.tsx correctly defers OneSignal init until after authentication. The pushNotificationService.initialize() is only called via registerUser() which is gated behind user?.type === 'locksmith'. This matches the iOS fix and should prevent the startup crash.

✅ AAB File Integrity: The AAB passes ZIP integrity checks, contains all expected native libraries (arm64-v8a, armeabi-v7a, x86, x86_64), 6 DEX files, and the JS bundle.

✅ Security Fixes Verified: All three OneSignal API security vulnerabilities (BUG-001, BUG-002, BUG-003 from the security doc) are confirmed fixed on the production backend.

✅ API Error Handling: Login with invalid credentials, empty body, and injection attempts all return appropriate error messages. Network errors show user-friendly messages.

✅ Non-www Redirect Protection: The API client correctly uses www.locksafe.uk and has a guard to rewrite bare locksafe.uk URLs, preventing the 307 redirect issue that breaks POST requests.

Deployment Blockers (Must Fix)

  • Resolve OneSignal App ID mismatch (BUG-01): Determine which ID is canonical. If the AAB-embedded os_v2_app_... is correct, update all source files. If cd19d270-... is correct, rebuild the AAB. Either way, rebuild and re-verify.
  • Add auth guard to locksmith routes (BUG-02): Add a check in (locksmith)/_layout.tsx that redirects to login when user is null. This is a 5-line fix.
  • Add api_key.json to .gitignore (BUG-11): Prevent accidental commit of Apple API key.

Recommended Before Deployment

  • Implement the /api/auth/refresh endpoint on the backend, or remove the refresh logic from the mobile client to avoid silent failures.
  • Add server-side validation for password length (≥6 chars) and email format on the registration endpoint.
  • Wire up the "Forgot password?" link to a password reset flow or remove it from the UI.
  • Add authentication to POST /api/onesignal/subscribe to match the security posture of the other OneSignal endpoints.
  • Verify the AAB was built from the latest commit that includes the iOS Build 4 race condition fix — the stale iOS buildNumber (3 vs 4) suggests it may predate the fix.

Testing Limitations

Android-native testing was not possible in this environment — no Android emulator or physical device is available. Testing was performed via the Expo web export (React Native Web) and direct API testing against the production backend. The AAB was inspected at the binary level (manifest, embedded config, file structure). Key findings like the OneSignal App ID mismatch and missing auth guard are code-level issues that affect all platforms equally. Push notification runtime behaviour and native Android UI rendering could not be verified and should be tested on a physical device before deployment.
